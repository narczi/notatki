Git

Terminologia:


Repository (krótko: repo) – miejsce w którym przechowywany jest kod projektu. Inaczej zbiór wszystkich plików projektu.
Remote repository (krótko: remote) – kopia naszego lokalnego repozytorium, które znajduje się na jakimś serwerze.
Commit, to commit (czasownik) – zapisać w repozytorium zmiany dokonane w jakimś pliku.
Commit (rzeczownik) – zmiana wprowadzona w repozytorium kodu. Commit może być rozumiany jako migawka stanu całego projektu. Z takich właśnie migawek składa się proces powstawania oprogramowania w repozytorium git. Magia gita polega na tym, że możemy cofnąć się do dowolnej migawki z przeszłości i przywrócić wcześniejszą wersję naszego projektu.
Commit hash – unikalny identyfikator dla danego commita.
Branch (gałąź) – jest to odrębna kopia całego projektu na której możemy dokonywać zmian nie martwiąc się przy tym, że zepsujemy coś na głównej gałęzi (pniu). Projekt może mieć dowolną ilość odgałęzień, ale główny kod zwykle znajduje się na gałęzi main. Gałęzie tworzy się gdy chcemy poeksperymentować bez naruszania głównej gałęzi. Osobne gałęzie można tworzyć także dla pojedynczych funkcjonalności projektu i scalać je z główną gałęzią dopiero, gdy będą w pełni gotowe.
Push – wysyłanie (pchanie) zmienionych plików do repozytorium.
Pull -

Pull request – prośba o scalenie naszego kodu z główną gałęzią, taka prośba może brzmieć np. tak: „will you please take my changes and pull them into master?” Jest to prośba o wciągnięcie (pull) naszych zmian do głównego brancha, inaczej zmergowanie. W gicie wysyłanie zwykło określać się mianem „pchania” (push), a odbieranie „wciąganiem” (pull). Nie możemy na siłę wypchnąć naszych zmian do innego brancha, możemy jedynie poprosić o wciągnięcie ich przez kogoś kto zarządza docelowym branchem.
Merge – scalenie wielu gałęzi w jedną.
Fork, to fork (czasownik) – utworzyć kopię czyjegoś repozytorium w celu pracy na nim. Dzięki forkowaniu nie potrzebujemy uprawnień do pracy na czyimś repozytorium, możemy pracować na kopii i dopiero gdy nasze zmiany będą gotowe – poprosić o zmergowanie ich.
Clone, to clone (czasownik) – klonowanie to czynność polegająca na skopiowaniu repozytorium, które znajduje się w internecie na nasz lokalny dysk.
origin – domyślna nazwa zdalnego repozytorium.
Staging area – miejsce do tymczasowego przechowywania plików, zanim zostaną dodane do commita (zostaną zacommitowane).

Komendy:

Konfiguracja:

Wyświetl zapisaną konfigurację:

git config --list

Skonfiguruj lokalnie:

git config user.email „mail@example.com”
git config user.name „johndoe”

Skonfiguruj globalnie:

git config --global user.email „mail@example.com”
git config --global user.name „johndoe”

Wyświetl status repozytorium:

git status

Wyświetl nazwę zdalnego repozytorium (najczęściej będzie to origin):

git remote

Wyświetl adres zdalnego repozytorium:

git remote -v

Zmiana remote repo:

git remote set-url origin git@github.com:<Username>/<Project>.git

Dodanie remote repo o nazwie „origin”:

git remote add origin https://github.com/narczi/django-dance.git

Stwórz nowe repozytorium git w istniejącym folderze:

git init

Branche:

Wyświetl lokalne gałęzie:

git branch

Wyświetl lokalne i zdalne gałęzie:

git branch -a

Stwórz nową gałąź:

git checkout -b
git checkout -b genericViews

Przełącz się do innej gałęzi:

git checkout genericViews

Zaktualizuj lokalną listę zdalnych branchy:

git remote update origin –prune

Praca z commitami:

Wyświetl commity:

git log
git log -2 (wyświetla dwa ostatnie commity)
git log -1 (analogicznie, wyświetla tylko ostatni commit)

Dodaj konkretny plik do commita:

git add plik.txt

Dodaj wszystkie pliki do commita

git add .

Usuń konkretny plik z commita:

git rm --cached plik.txt

Zacommituj (wszystkie) modyfikacje (dodaj do commita) wraz z wiadomością:

git commit -a -m „Opis commitu”

Wypchnięcie
(wysłanie) zmian do zdalnego repozytorium:



	
git push

	git push origin main
	(wypchnij do zdalnego repozytorium o nazwie „origin”, do gałęzi
	main)
